## TL;DR — дизайн или промпты?

**И то, и другое, но корень проблемы — баг в коде, из-за которого миссию
принципиально невозможно завершить.** Слои проблем, по убыванию критичности:

1. **🔴 Критический баг** — завершение миссии недостижимо (невозможно «дойти»
   ни за 10–15, ни за 100 ходов). Это чистый код, не промпт и не дизайн.
2. **🟡 Однообразие миссий** — проблема промпта генерации (нет каталога
   архетипов → LLM всегда выдаёт «перехвачен сигнал»).
3. **🟡 Выщелкивание экипажа по одному** — отсутствие механических ограничений
   на летальность и прогресс (весь баланс отдан одному LLM-вызову).

Таким образом, ответ на исходный вопрос: **сначала баг, затем промпт и дизайн**.
Промптами и геймдизайном можно вылечить пункты 2 и 3, но пункт 1 требует
правки кода.

## 1. Контекст — как устроен цикл хода

(см. `docs/RULES_RU.md`, раздел «Алгоритм каждого хода»)

1. Генерируется общая вводная дня (`build_daily_story_prompts`, `prompts.py:978`).
2. Для каждого игрока — персональный брифинг с 3–4 вариантами действий, у
   каждого **скрытое последствие** (`build_personal_briefing_system`, `prompts.py:1157`).
3. Живые игроки выбирают первыми; NPC выбирают вслепую после.
4. Один LLM-вызов `analyze_combined_outcome` (`game_master.py:~2290`) анализирует
   ВСЕ решения и возвращает структурированный исход: `mission_progress`,
   `dead_crew_members`, `crew_injured`, `ship_hull_integrity`, `ship_shields`,
   `personal_outcomes`, `next_day_hook` и т.д. (схема `COMBINED_OUTCOME_SCHEMA`,
   `prompts.py:395`).
5. Код применяет исход к данным (`main.py:~2480–2620`): копит прогресс этапов,
   помечает погибших, пересчитывает `crew_health`, проверяет завершение миссии.

Ключевой архитектурный факт: **все решения о балансе (сколько прогресса,
кого убить, насколько пробить корпус) принимает LLM в одном вызове.**
Детерминированных ограничений (кэпы, лимиты, темп) в коде нет.

## 2. Находка №1 (критично) — миссию невозможно завершить

### 2.1. Доказательство из БД

Единственная запущенная игра (`default_game`):

```
Миссия:     «Призрачный сигнал: Протокол Эхо Ориона»   ← снова «сигнал»
objectives: 3 этапа
stage_progress: {"1": 6, "2": 6, "3": 2}                ← прогресс копится
current_stage: 0
total_stages: 1
completed:   0
game_state.day: 3, crew_health: 96
```

На этапах 1 и 2 уже по 6 очков (выше любого разумного порога), но
`current_stage = 0`, `total_stages = 1`, `completed = 0`. Миссия застряла
навсегда.

### 2.2. Три связанных дефекта в коде

**Дефект A — поля `current_stage`/`total_stages` никогда не задаются.**
`MISSION_SCHEMA` (`game_master.py:600`) содержит только `[name, description,
objectives]` — полей `current_stage`/`total_stages` в схеме **нет**.
`generate_mission` (`game_master.py:2372`) возвращает результат LLM как есть
(и fallback-ветка тоже их не задаёт). Далее `create_mission`
(`database.py:1744`) сохраняет:

```python
mission_data.get("current_stage", 0)   # → всегда 0
mission_data.get("total_stages", 1)    # → всегда 1
```

Поэтому **любая** миссия рождается с `current_stage=0`, `total_stages=1`,
независимо от числа этапов.

**Дефект B — off-by-one в проверке прохождения этапа.**
`main.py:2517`:

```python
if points_int > 0 and stage_num == current_stage:   # current_stage=0
```

Этапы в `objectives` **1-индексированы** (`stage: 1, 2, 3`; подтверждено схемой
и fallback-веткой `"stage": i + 1`). Условие `stage_num == 0` **никогда** не
выполняется → `current_stage` навсегда остаётся 0 → ни один этап не
«завершается».

**Дефект C — ложное завершение при наивном фиксе.** Даже если завести
`current_stage = 1`, `total_stages = N`, текущая логика сломана иначе:

```python
current_stage = min(current_stage + 1, total_stages)   # крышка на N
...
completed = current_stage >= total_stages              # N >= N
```

Завершив этап N-1, `current_stage` становится N (= `total_stages`), и
`completed = N >= N` срабатывает **до** того, как этап N достигнут. То есть
наивный фикс дефектов A+B автоматически породит ложное «миссия пройдена» при
переходе к последнему этапу.

> В сегодняшний день дефект C маскируется дефектами A+B (`current_stage` застрял
> на 0), поэтому в проде наблюдается «никогда не завершается». Любой фикс
> должен учитывать все три.

### 2.3. Рекомендуемое исправление (робастное)

1. В `generate_mission` (`game_master.py:2372`) нормализовать результат перед
   возвратом (и для основного, и для fallback-пути):

   ```python
   objectives = sorted(result["objectives"], key=lambda o: o["stage"])
   for i, o in enumerate(objectives):
       o["stage"] = i + 1                                  # строгая 1-индексация
       o["success_threshold"] = clamp(o["success_threshold"], 3, 5)  # новая политика
   result["objectives"] = objectives
   result["current_stage"] = 1
   result["total_stages"] = len(objectives)
   ```

2. В `create_mission` (`database.py:1744`) — на всякий случай выводить
   `total_stages` из `len(objectives)`, если поле не пришло.
3. Заменить хрупкую проверку завершения на явную (иммунную к индексации):

   ```python
   completed = all(
       stage_progress.get(str(o["stage"]), 0) >= o["success_threshold"]
       for o in mission["objectives"]
   )
   ```

   А продвижение `current_stage` вычислять как «индекс первого незавершённого
   этапа» (а не инкрементом с крышкой):

   ```python
   current_stage = next(
       (o["stage"] for o in objectives
        if stage_progress.get(str(o["stage"]),0) < o["success_threshold"]),
       total_stages + 1,
   )
   ```

   Так `current_stage` корректно указывает на текущий рабочий этап, а
   `completed` определяется реальным достижением всех порогов.

## 3. Находка №2 — однообразие миссий («опять сигнал»)

### 3.1. Причина

`build_mission_prompts` (`prompts.py:1043`) — 6 строк без какой-либо
вариативности: «создай миссию с названием, описанием, 2–4 этапами». Нет
каталога архетипов, нет запретов на клише, нет разнообразия сеттингов. LLM без
ограничений всегда выбирает самый затёртый троп — «перехвачен сигнал / SOS,
летим разбираться». Миссия в БД буквально так и называется.

### 3.2. Предлагаемое решение — инъекция архетипа и таблиц (random tables)

LLM **без ограничений — банален, с ограничениями — креативен.** Идея:
детерминированно выбрать «зерно» миссии и inject'нуть его в промпт.

**A. Кураторский каталог архетипов** (выбирается случайно на игру):
`first_contact` (дипломатия/первый контакт), `rescue` (спасение/эвакуация),
`survey` (научная разведка), `mystery` (детектив/тайна), `infiltration`
(проникновение/кража), `defense` (оборона/бой), `intrigue` (политическая
интрига), `trade` (торговля/переговоры), `anomaly` (изучение аномалии),
`exploration` (исследование неизведанного). Каждый архетип несёт: типичную
структуру этапов, рекомендуемые осложнения, тон.

**B. Random-таблицы** (подмешиваются по одной записи в промпт):

- **Сеттинг/локация:** поверхность планеты, орбитальная станция, туманность,
  руины древней цивилизации, дереликт, горизонт событий, газовый гигант,
  астероидное поле…
- **Антагонист/осложнение:** природное явление, взбунтовавшийся ИИ, пираты,
  зараза, политическая фракция, временная аномалия, конкурирующая экспедиция,
  внутренний конфликт экипажа…
- **Твист** (поворот на 2–3 ходе): «союзник — предатель», «сигнал — из
  будущего», «объект — живой», «цель меняется», «награда имеет цену»…
- **Награда:** технология, артефакт, союзник, карта, данные, ресурс, репутация.

**C. Список запрещённых завязок:** в промпте явно запретить «перехвачен сигнал /
SOS / неопознанная передача / аномальное излучение» как затравку миссии.

**D. Хранение зерна:** добавить в `game_missions` колонку `archetype` + JSON
`seeds` (через миграцию — см. §6), чтобы архетип и таблицы были стабильны на
протяжении всей миссии и учитывались в каждом ходе.

## 4. Находка №3 — летальность и «выщелкивание по одному»

### 4.1. Причины

- **Смерть бинарна** (жив/мёртв), нет HP-буфера. LLM в `dead_crew_members`
  просто решает «этот умер» → код вызывает `mark_player_dead`
  (`main.py:~2578`). Никакого ограничения «макс N смертей за ход/игру» нет.
- **Регресс свободный:** `mission_progress` может быть `−M` любой величины
  (`prompts.py:618` — «points: +/-M»). Один «неправильный» выбор стирает
  прогресс нескольких ходов. Код лишь ограничивает снизу нулём
  (`new_progress = max(0, old + points)`, `main.py:2510`).
- **Промпт требует драмы любой ценой:** «Must be DRAMATIC», «Every turn MUST
  CHANGE something» (`prompts.py:609`, `:583`). LLM фабрикует «перемены» через
  урон/смерти/регресс — это становится нормой.
- `crew_health` в БД — это **здоровье корабля**, не экипажа: считается как
  `hull*0.7 + shields*0.3` (`main.py:2545`). Индивидуального здоровья членов
  экипажа нет вовсе — только флаг `is_dead`.
- NPC выбирают вслепую и с весом NORMAL, но могут тянуть весь экипаж вниз.

### 4.2. Предлагаемое решение

**A. Многоуровневое здоровье экипажа** (опционально, основной рычаг):
ввести состояния `healthy → light → serious → critical → dead`, персистить в
БД. Смерть — только от накопленного урона, а не единичного декрета LLM.
Ранения **лечатся** со временем (появляется смысл у роли медика). Это требует
миграции (§6).

**B. Упрощённая альтернатива** (без полной HP-системы, меньше работы):
жёсткие лимиты на уровне кода, применяемые ПОСЛЕ ответа LLM:

- Максимум **1 смерть на N ходов** (например, 1 на 4 хода) на игру; LLM может
  «предлагать» больше — код отклоняет лишнее и понижает до ранения.
- Никогда не убивать ниже минимального живого состава без явного
  «финального» контекста.
- Каждая «смерть» сверх лимита автоматически становится `critical` ранением.

**C. Переписать промпт `combined_outcome`** (`prompts.py:540–630`):

- Убрать «Must be DRAMATIC / MUST CHANGE something» как двигатель наказаний.
- Переформулировать: драма приходит через **открытия/твисты/осложнения**, а не
  через трупы. «Каждый ход что-то меняет, но перемена ≠ урон».
- Явно: ранения/смерти — редкое, кумулятивное следствие **явно рискованных**
  действий, а не дефолтная «драма».

**D. Кэпы прогресса** (слой правил, см. §5):

- Регресс **только** для явно рискованных/ошибочных выборов, **макс −1**.
- Уже пройденные этапы **не откатываются**.
- Каждый недраматичный ход даёт **≥ +1** к текущему этапу (гарантия
  продвижения).

**E. Защитная агентность:** в персональный брифинг добавить надежное действие
«прикрыть/защититься/эвакуироваться», предсказуемо снижающее урон.

**F. NPC — нейтрально-положительные:** решения NPC должны быть
компетентными по роли (не источником просадки). Либо математически снизить их
отрицательный вклад до 0 (кодом), либо в промпте указать «NPC действует
грамотно и редко вредит».

## 5. Архитектурная идея — «слой правил» между LLM и БД

Корень пунктов 3–4: сейчас LLM делает **и рассказ, и баланс** одновременно.
Предложение — тонкий детерминированный слой, который принимает «сырой» ответ
LLM и применяет его по правилам:

```
LLM (combined_outcome): предлагает mission_progress[], deaths[], injuries[], hull…
        │
        ▼
Слой правил (НОВЫЙ, в коде):
  - clamp(thresholds 3–5), гарантия ≥+1 прогресса на недраматичный ход
  - кэп регресса (max −1, без отката пройденных этапов)
  - лимит смертей за ход/игру, пересыл «лишних» в ранения
  - применение HP/ранений, лечение со временем
  - вычисление completed по реальным порогам (см. §2.3)
        │
        ▼
БД (через существующие функции database.py)
```

LLM остаётся креативным (нарратив, твисты, персональные исходы), движок
гарантирует честность. Этот слой — то, чего сейчас физически нет.

## 6. Изменения модели данных (только через `MIGRATIONS`)

> Согласно `AGENTS.md`: любые изменения схемы — только новыми записями в списке
> `MIGRATIONS` (`database.py`), никогда правкой существующих `CREATE TABLE`.

Предлагаемые миграции (формат соответствует существующим записям):

   ```sql
   ALTER TABLE game_missions ADD COLUMN archetype TEXT DEFAULT '';
   ALTER TABLE game_missions ADD COLUMN seeds TEXT DEFAULT '{}';
   ```

2. **Здоровье экипажа** (для §4.2A, если берётся HP-вариант):

   ```sql
   ALTER TABLE player_profiles ADD COLUMN health_state TEXT DEFAULT 'healthy';
   ALTER TABLE npc_profiles     ADD COLUMN health_state TEXT DEFAULT 'healthy';
   ALTER TABLE game_state       ADD COLUMN last_death_day INTEGER DEFAULT 0;
   ```

   `health_state ∈ {healthy, light, serious, critical, dead}`. Лечение:
   переход на уровень вверх каждые K ходов (роль медика ускоряет).
3. **(Опц.) Перехват смертей:** таблица `crew_injuries` для истории ранений,
   если нужна детальная механика лечения.

Упрощённый вариант (§4.2B) требует только миграцию 3-го пункта
(`last_death_day`), без полной HP-системы.

## 8. Фазирование и приоритеты

| Фаза | Что | Эффект | Объём |
|------|-----|--------|-------|
| **P0** | Фикс бага завершения миссии (§2.3) — три дефекта + пороги 3–5 | Миссию **вообще можно** пройти за 6–12 ходов | Малый |
| **P1** | Кэпы прогресса/регресса + гарантия ≥+1 (§4.2D, §5) | Убирает «нудятину» и вечное стояние | Малый–средний |
| **P2** | Каталог архетипов + random-таблицы (§3) | Убирает «опять сигнал» | Средний |
| **P3** | Рework промпта combined_outcome + лимиты смертей (§4.2B,C,F) | Убирает «выщелкивание по одному» | Средний |
| **P4** | Полная HP/ранения-система + лечение (§4.2A, миграции) | Глубина, роль медика | Большой (опционально) |

P0+P1 уже превратят игру из «невозможно пройти» в «проходимо и динамично».
P2–P3 снимут повторяемость и избыточную летальность. P4 — качественное
улучшение, можно отложить.

## 9. Риски и открытые вопросы

- **Точность фикса P0:** дефект C (ложное завершение) легко пропустить —
  обязательно покрыть тестом на миссии из 2 и 3 этапов (включая граничный
  случай «последний этап не достигнут»).
- **Зерно архетипа:** как выбирать (равномерный рандом / взвешенно / по
  составу экипажа)? Начать с равномерного, уточнить потом.
- **HP-система vs лимиты:** полная HP (P4) — большой объём; лимиты смертей
  (P3) дают 80% эффекта за 20% работы. Рекомендую сначала лимиты.
- **NPC-баланс:** снижение негативного вклада NPC может сделать игру слишком
  лёгкой — нужен калибровочный прогон.
- **Совместимость с существующей игрой:** в БД уже есть миссия с
  `current_stage=0, total_stages=1` — после фикса её нужно либо
  ретроспективно нормализовать (вычислить `current_stage`/`total_stages` из
  `objectives` и `stage_progress`), либо пересоздать.
