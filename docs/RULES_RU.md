# Правила AI Game Master

## Обзор игры

AI Game Master — это кооперативная игра с генерируемым искусственным интеллектом сюжетом,
доставляемая через Telegram-бот. Каждый ход генерируется уникальная история в космическом
сеттинге (звёздный корабль, экипаж, приключения), где игроки принимают решения,
влияющие на развитие сюжета.

## Основные механики

### 1. Регистрация и онбординг

- После `/start` игрок проходит интервью (5+ вопросов с вариантами)
- Вопросы генерируются LLM, каждый вариант содержит очки (role_scores) для ролей
- Дополнительные вопросы определяют вид (species) и пол (gender) персонажа:
  - 10 вопросов о виде (человек, гуманоид, негуманоид, энергия, кибернетика, симбиот)
  - 4 вопроса о поле (мужской, женский, нейтральный, сменяемый, множественный и т.д.)
- На основе ответов через ComfyUI генерируется аватар с учётом вида/пола
- Роль назначается детерминированно по максимальной сумме очков (role_scores)
- После онбординга игрок получает роль, описание, вид, пол, аватар

### 2. Формирование экипажа

- **GAME_START_MIN_PLAYERS** (по умолчанию 3) — минимальное количество живых игроков для старта
- Пока экипаж не набран, свободные роли занимаются **NPC**
- Живой игрок, завершивший онбординг, может **заменить NPC** в той же роли
- NPC освобождает роль, если живой игрок получает ту же роль
- При завершении онбординга всем остальным игрокам рассылается **уведомление с аватаром и профилем**

### 3. Старт игры и миссия

При старте игры (когда набрано GAME_START_MIN_PLAYERS живых игроков):

1. **Генерация NPC** — для всех свободных ролей создаются NPC с аватарами:
   - Без онбординга и интервью
   - Вид (species) и пол (gender) выбираются случайно
   - Prompt для аватара генерируется рандомно для разнообразия
2. **Генерация миссии** — LLM создаёт:
   - Название и описание миссии
   - Цели миссии с разбивкой на **этапы** (stage-based progression)
   - Каждый этап имеет требования для завершения
   - Данные миссии хранятся отдельно (таблица `game_missions`)
   - Учитываются при генерации каждого следующего хода
3. **Генерация картинки рубки** — сложный image pipeline:
   - LLM генерирует промпт с учётом всех ролей и аватаров
   - Аватары команды используются как **reference изображения** (ControlNet/IP-Adapter)
   - Единая сцена: мостик корабля с командой на своих постах
4. **Брифинг миссии** — всем игрокам рассылается:
   - Описание миссии и целей
   - Сгенерированная картинка рубки с командой

## Алгоритм каждого хода (turn)

> Внутренняя нумерация в БД и API использует `day`, но в игровой механике и
> пользовательском интерфейсе используется термин **turn** (ход).

```text
┌─────────────────────────────────────────────────────────────────┐
│  НАЧАЛО ХОДА                                                   │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  1. Генерация общей вводной на основе предыдущего хода:        │
│     - Если есть предыдущий ход — учитываются последствия       │
│     - Учитывается текущее состояние миссии и этапы             │
│     - Создаётся описание ситуации на корабле и вокруг          │
│                                                                 │
│  2. Генерация промпта для картинки по общей вводной           │
│     → Генерация картинки через ComfyUI                         │
│                                                                 │
│  3. Генерация брифинга для КАЖДОГО игрока:                    │
│     - Персональная вводная (что видит/слышит этот персонаж)   │
│     - 3-4 варианта действий                                    │
│     - У каждого действия есть СКРЫТОЕ последствие              │
│     - Часть последствий ухудшает состояние, часть улучшает     │
│     - Последствия имеют коэффициенты влияния (запоминаются)    │
│                                                                 │
│  4. Генерация промпта индивидуальной картинки брифинга        │
│     → Генерация картинки через ComfyUI                         │
│                                                                 │
│  5. Отправка живым игрокам:                                    │
│     - Общая вводная + общая картинка                           │
│     - Индивидуальный брифинг + индивидуальная картинка         │
│     - Варианты действий                                        │
│                                                                 │
│  6. Сбор ответов от живых игроков                              │
│                                                                 │
│  7. Как только ВСЕ живые игроки ответили:                     │
│     - Вводные передаются NPC                                   │
│     - NPC выбирают действия НЕ ЗНАЯ последствий                │
│     - NPC не видят выборов других игроков                      │
│                                                                 │
│  8. Анализ ВСЕХ выбранных последствий:                        │
│     - Генерация общих последствий (combined outcome)           │
│     - Изменение состояния команды и корабля                    │
│     - Действия игроков имеют БОЛЬШИЙ вес, чем NPC             │
│     - Проверка достижения целей миссии                         │
│       → Этапы могут достигаться НЕЛИНЕЙНО, эффект накапливается│
│     - Если цели достигнуты → оповещение живых игроков          │
│                                                                 │
│  9. Допускается:                                               │
│     - Гибель части членов экипажа                              │
│     - Гибель корабля (история не бесконечная)                  │
│                                                                 │
│ 10. Для мёртвых членов экипажа (не NPC):                      │
│     - На каждом ходу генерируются последствия и вводные        │
│     - Они остаются зрителями (видят, но не влияют)             │
│     - Мёртвый игрок может нажать /start                        │
│       → присоединиться к новой игре                            │
│       → или к текущей в новой роли (заменяя NPC)               │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**Ключевые принципы:**

- Действия игрока влияют на последствия **сильнее**, чем действия NPC
- Прогресс миссии нелинейный — правильные действия накапливают прогресс
- Состояние корабля и экипажа отслеживается числовыми коэффициентами
- Концовка возможна: гибель корабля или успешное завершение миссии

## Поведение Game Master (корабельный компьютер)

### При начале нового хода

1. **Проверка состояния игры**
   - Проверяет, что игра не закончена (корабль и экипаж живы)
   - Проверяет, все ли живые игроки ответили
   - Если игра закончена — генерация прекращается

2. **Генерация сюжета**
   - Учитывает последствия предыдущего хода
   - Учитывает прогресс миссии и этапов
   - Обеспечивает связность и последовательность истории

3. **Генерация контента**
   - Общая картинка через ComfyUI
   - Индивидуальные картинки брифингов

4. **Обработка действий**
   - Живые игроки выбирают первыми
   - NPC выбирают после (без знания последствий)
   - Анализ последствий с учётом веса игроков

## Взаимодействие с игроками

### Текстовые сообщения

- Игроки могут общаться с Game Master в любое время
- Game Master отвечает в характере корабельного компьютера

### Голосовые сообщения

- Поддержка голосовых сообщений (без транскрибации)
- Запись и хранение

## Игровые механики

### Действия и последствия

- Каждое действие имеет видимый текст и **скрытое последствие**
- Последствия имеют числовые коэффициенты влияния
- Часть последствий улучшает состояние, часть ухудшает
- Система запоминает все коэффициенты для каждого действия

### Прогресс миссии

- Миссия разбита на этапы (stages)
- Этапы достигаются нелинейно — правильные действия накапливают прогресс
- Прогресс отслеживается числовым счётчиком
- При достижении порога этап считается пройденным

### Командная игра

- Решения всех игроков влияют на общий исход
- Действия живых игроков имеют больший вес
- NPC действуют логично в рамках своей роли

### Смерть и зрительство

- Члены экипажа могут погибнуть
- Мёртвые не-NPC игроки остаются зрителями
- Зрители видят развитие сюжета, но не влияют
- Могут переприсоединиться через /start

## Технические детали

### Архитектура системы

```text
┌─────────────────────────────────────────────────────────┐
│  Telegram Bot (aiogram)                                │
│  - Команды: /start, /profile, /today, /help            │
│  - Онбординг с FSM                                     │
│  - Обработка сообщений                                 │
└──────────────────────┬──────────────────────────────────┘
                       │
                       ▼
┌─────────────────────────────────────────────────────────┐
│  Game Master API (FastAPI)                             │
│  - Генерация сюжета через LLM                         │
│  - Генерация изображений через ComfyUI                 │
│  - Управление состоянием                               │
└──────────────────────┬──────────────────────────────────┘
                       │
               ┌───────┼───────┐
               ▼       ▼
        ┌─────────┐ ┌──────────┐
        │Scheduler│ │ComfyUI   │
        │(cron)   │ │(GPU gen) │
        └─────────┘ └──────────┘
```

### База данных

- SQLite для хранения профилей, сессий, ходов, действий, сообщений, миссий
- Миссии хранятся отдельно (game_missions, mission_stages)
- Каждый ход привязан к game_id и day (внутренняя нумерация)

## Заключение

AI Game Master создаёт уникальный кооперативный опыт с живой историей,
генерируемым контентом и глубокими механиками последствий.
Игроки не просто выбирают действия — они определяют судьбу экипажа и корабля.
